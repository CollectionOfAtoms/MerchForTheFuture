As of Epic MFTF-12 it is also wired as the `DESIGNED` apparel path — but mostly as a **reference integration** to prove the multi-provider abstraction: Prodigi has **no organic / 100%-cotton apparel option**, so it won't serve production apparel (which is GOTS-certified Teemill). ## Summary for MFTF-3 / MFTF-12

- **Base URL:** `https://api.prodigi.com/v4.0` (env `PRODIGI_API_BASE_URL`). **Sandbox exists:** `https://api.sandbox.prodigi.com/v4.0` — unlike Teemill, Prodigi can be exercised without a real charge using a sandbox key.
- **Auth:** header `X-API-Key: {PRODIGI_API_KEY}`. No `project` param (that's a Teemill thing).
- **Currency:** Quotes accept `currencyCode` and return amounts in it — we request **USD**, so (unlike Teemill's GBP) **no FX conversion** is needed for the buyer total.
- **No catalog-listing endpoint** — see below.

## No catalog-listing endpoint (discovery is manual)

Prodigi's v4 API has **no "list all products" / search endpoint**. You can only fetch a product you already know the SKU for (`GET /v4.0/products/{sku}`). Discovery of *which* SKUs exist is via Prodigi's **downloadable product catalogue** (their dashboard / website), not the API. Consequences in this codebase:

- **Prints:** `src/lib/print/listing.ts` holds a **verified static `PRINT_CATALOG`** (FAP + CAN SKUs with dimensions), generated by brute-force-probing candidate SKUs with `scripts/probe-prodigi-catalog.ts` and keeping the 200s. Re-run that script after any Prodigi catalog change.
- **Apparel:** there is no bulk fetch, so the size/colour sync (`src/lib/apparel/sync-prodigi.ts`) enumerates our **own** designed `ProductType.providerSkuBase` values and GETs each — no manual SKU list at sync time, but the SKUs themselves still come from Prodigi's catalogue when an admin creates the product type.

## `GET /v4.0/products/{sku}` — product details (live-verified 2026-06-17)

Returns `{ product: { sku, description, productDimensions?, variants: [...] } }`.

- **Prints:** `productDimensions: { width, height, units }` (the SKU encodes the canonical inch size, e.g. `GLOBAL-FAP-16X24`).
- **Apparel variants:** each variant is
  ```json
  {
    "attributes": { "brand": "Bella + Canvas", "edge": "Crew neck",
      "color": "navy blue", "gender": "Women's",
      "paperType": "52% cotton, 48% polyester", "size": "m", "style": "1010" },
    "shipsTo": ["US"],
    "printAreaSizes": { "back": { "horizontalResolution": 2490, "verticalResolution": 3510 },
                        "front": { "horizontalResolution": 2490, "verticalResolution": 3510 } },
    "pagePricing": null
  }
  ```
- **Colours:** lowercase **name strings only** in `variants[].attributes.color` (e.g. `"black"`, `"navy blue"`, `"baby blue"`, `"natural"`). **No hex, no colour code, no swatch image.** We derive a display hex from the name (`src/lib/apparel/color-hex.ts`, `colorNameToHex`) since the UI swatch keys off a hex.
- **Sizes:** lowercase in `variants[].attributes.size` (`xs`,`s`,`m`,`l`,`xl`,`2xl`), in **arbitrary order**. We canonicalise + sort with the size enum (`src/lib/apparel/sizes.ts`, `canonicalSizeLabel`/`sizeRank`/`normalizeSizes`): `m`→`M`, `2xl`→`XXL`, smallest→largest.
- **`paperType`** is the fabric composition (e.g. `"52% cotton, 48% polyester"`). This is the field a future cotton-standard guard would read.
- **`printAreaSizes`** keys are the available print areas (`front`/`back`). Our designed listings put the single design on **`front`**.

## `POST /v4.0/quotes` — shipping + price quote (live-verified 2026-06-17)

Request:
```json
{
  "currencyCode": "USD",
  "destinationCountryCode": "US",
  "shippingMethod": "Standard",   // OPTIONAL — omit to get a quote per service tier
  "items": [
    { "sku": "...", "copies": 1, "attributes": { "size": "2xl", "color": "white" },
      "assets": [{ "printArea": "front" }] }
  ]
}
```

- **Apparel items REQUIRE** `attributes: { size, color }` (raw Prodigi spelling) **and** `assets: [{ printArea: "front" }]`. Omitting them → **HTTP 400** with `{ "outcome": "ValidationFailed", "failures": { "items[n].attributes": [{ "code": "MissingRequiredAttributes", ... }], "items[n].assets": [{ "code": "MissingRequiredAssets"/"UnexpectedAssets", ... }] } }`. The `validValues` in the failure conveniently list the accepted sizes/colours.
- **Prints** carry **no attributes** and use `assets: [{ printArea: "default" }]` (the SKU encodes the size).
- **Omitting `shippingMethod`** returns a quote for **every available service tier** (Budget / Standard / Express / Overnight) in `quotes[]` — this powers buyer-selectable shipping (`12.3`). // UNVERIFIED that omission returns *all* tiers — observed behaviour, confirm on a live multi-tier quote.
- Response: `{ "quotes": [ { "shipmentMethod": "Budget", "costSummary": { "shipping": { "amount": "3.50", "currency": "USD" } }, ... } ] }`. We read `quotes[].costSummary.shipping.amount`.

## `POST /v4.0/orders` — order submission (single-step)

Unlike Teemill's two-step (create → confirm), Prodigi is **single-step**. Request:
```json
{
  "shippingMethod": "Standard",
  "recipient": { "name": "...", "address": { "line1", "townOrCity", "stateOrCounty", "postalOrZipCode", "countryCode" } },
  "items": [
    { "sku": "...", "copies": 1, "sizing": "fillPrintArea",
      "attributes": { "size": "2xl", "color": "white" },        // designed apparel only
      "assets": [{ "printArea": "front", "url": "https://…/design.png" }] }
  ]
}
```
- Designed apparel → `attributes` + `assets[{ printArea: "front", url: designImageUrl }]`.
- Prints → no attributes, `assets[{ printArea: "default", url: printSourceImageUrl }]`.
- `sizing: "fillPrintArea"` is carried from the print path. // UNVERIFIED whether `fitPrintArea` is preferable for logo-style apparel designs (fill may crop) — evaluate on the proof print.
- Response: `{ "order": { "id": "...", "status": { "stage": "..." }, "shipments": [...] } }`. We store `order.id` as `FulfillmentOrder.providerOrderId`.
- **NOT yet run live for designed apparel** — deferred to the first design-proof order.

## Canvas edge wrap + image `sizing` (LIVE-VERIFIED 2026-06-21, sandbox)

Stretched-canvas SKUs (`GLOBAL-CAN-*`) have physical sides, and Prodigi controls
how they're finished **per line item**. Both confirmed against `api.sandbox.prodigi.com`
(`scripts/probe-prodigi-canvas-attributes.ts`; sandbox orders `ord_1161006`/`ord_1161007`):

- **Edge wrap → item `attributes.wrap`.** `GET /products/GLOBAL-CAN-{size}` returns
  `product.attributes.wrap = ["Black", "ImageWrap", "MirrorWrap", "White"]` (also
  `edge:"38mm"`, `frame:"38mm standard stretcher bar"`, `paperType:"Standard canvas (SC)"`,
  `substrateWeight:"400gsm"` — all single-valued). `ImageWrap` continues the image around
  the sides; `MirrorWrap` mirrors it; `Black`/`White` are solid edges.
  - Accepted on **both** `POST /quotes` and `POST /orders` as `items[].attributes.wrap`, and
    echoed on the created order's `items[].attributes`.
  - An invalid value → **`400 ValidationFailed`** with
    `items[0].attributes[0] → { code: "MustBeInAllowedValues", allowedValues: { name: "wrap", values: [...] } }`.
  - Paper prints (`GLOBAL-FAP-*`) have **no** `wrap` attribute (no edges) — sending one would 400.
- **Image fit → item `sizing` (ORDER-ONLY).** `sizing` lives on `POST /orders` items
  (`fillPrintArea` scales to fill+crop — our current default; `fitPrintArea` fits the whole
  image, possibly with margins; `stretchToPrintArea` distorts). It is **rejected on `/quotes`**
  (`ModelBindingFailed → items[0].sizing: UnknownField`) — quotes take only `attributes`.
- **SKU existence:** `GET /products/{sku}` is **200** for a real SKU, non-200 otherwise — this is
  the authoritative way to validate any SKU (apparel blank or print) before persisting it.

> **Implemented in Epic MFTF-PF (2026-06-21).** `src/lib/fulfillment/providers/prodigi.ts`
> now sends, per print line item:
> - **`attributes.wrap`** for canvas items (`GLOBAL-CAN-*`), mapped from the seller's
>   stored `PrintFraming.wrap` (default `MirrorWrap`; `ImageWrap` excluded from the picker).
>   Paper items (`GLOBAL-FAP-*`) send **no** `wrap`. This rides in the item `attributes`, so
>   it appears on **both** `POST /orders` and `POST /quotes` (parity).
> - **`sizing: "fitPrintArea"`** for **framed** items (the asset is the seller's exact-aspect
>   crop, so we want the whole confirmed crop shown with no fill-crop of the face). The
>   resolver tags framed items with `framed: true`; the chosen value is `fitPrintArea`.
>   **Apparel and the defensive fallback path keep `fillPrintArea`** (unchanged). `sizing` is
>   order-only, so quotes are unaffected.
> - The **asset URL** is the framed `PrintFraming.croppedUrl` (production file), not the raw
>   source. Defensive fallback (unreachable for an ACTIVE listing post-gate): no framing row →
>   original source image + (canvas) `MirrorWrap`, logged as an anomaly (`[print-fanout] ANOMALY`).
>
> **// UNVERIFIED — rotated framing orientation.** The framing tool (US-MFTF-PF.3) lets a seller
> rotate the crop box 90°, so a landscape piece can be framed landscape on a portrait-named SKU
> (e.g. an exported `croppedUrl` with a ~17:11 pixel aspect sent for `GLOBAL-CAN-11X17`). The
> `PrintFraming` row is still keyed by the SKU's nominal aspect; only the produced image's
> orientation differs. **It is not yet confirmed whether Prodigi auto-rotates the asset to the
> print area or expects it pre-oriented.** With `fitPrintArea` the whole crop should be shown
> (rotated or letterboxed), but confirm on the first live order that a rotated crop prints
> correctly — if Prodigi does not auto-rotate, we'd need an explicit orientation field or to
> rotate the asset server-side before upload.

## `GET /v4.0/orders/{id}` — status & tracking

Response `{ "order": { "status": { "stage": "..." }, "shipments": [{ "tracking": { "number", "carrier" } }] } }`.

- `getOrderStatus` maps `stage`: `InProgress`→PROCESSING, `Complete`→(SHIPPED if shipments else DELIVERED), `Cancelled`→CANCELLED, else ERROR.
- `checkFulfillmentStatus` (12.6) treats `stage` `Complete`/`Dispatched`/`Shipped` **with** `shipments[0].tracking.number` as shipped, returning `{ shipped, trackingNumber, carrier }`. Prodigi exposes the same data its webhook carries, so the polling read and the webhook share one status contract (Teemill, by contrast, only polls).

## Webhooks (US-MFTF-14.1) — `// UNVERIFIED`

Prodigi supports order-status webhooks (CloudEvents-style callbacks), so unlike Teemill it is on
the **webhook** path, not polling. The route is `POST /api/webhooks/prodigi`
(`src/app/api/webhooks/prodigi/route.ts`). The auth model is **confirmed** (2026-06-19); the event
names + payload field paths are still `// UNVERIFIED` (no live callback delivered to this account
yet) — confirm and tighten on the first real dispatch.

- **Authenticity (CONFIRMED 2026-06-19):** Prodigi does **not** sign callback payloads and does
  **not** issue a signing secret. You only register a callback URL and the body arrives as an
  **unsigned CloudEvents** payload (no signature header). We secure it with a **per-order token**:
  the order object accepts a `callbackUrl`, so at fan-out time (`src/lib/checkout/fanout.ts`) each
  Prodigi `FulfillmentOrder` mints an unguessable `webhookToken` and registers
  `${NEXT_PUBLIC_BASE_URL}/api/webhooks/prodigi?token=<webhookToken>` on the order. Prodigi POSTs
  status callbacks to that exact URL, so the route resolves the shipment by `webhookToken` alone
  (`findFirst({ where: { provider: "prodigi", webhookToken } })`): a token matching no shipment →
  **401**. No global dashboard callback URL is configured; **no shared secret env var** is needed.
  Wins: a leaked token compromises one order only, and each environment self-addresses its own
  callbacks (a preview/dev order's callbacks return to that host, not prod). Defence in depth: past
  the token, only enumerated event types act. (The polling cron remains the backstop if a callback
  is ever missed.)
- **Handled event types** (the enumerated set in `HANDLED_PRODIGI_EVENTS`,
  `src/lib/fulfillment/providers/prodigi.ts`). Anything outside this set is acknowledged **200 and
  ignored** (no transition), so unexpected events never retry-storm:

  | Event `type` | Canonical status | Notes |
  |---|---|---|
  | `com.prodigi.order.status.stage.changed#InProgress` | `PROCESSING` | no-op from CONFIRMED |
  | `com.prodigi.order.status.details.printStatus#Printing` | `PRINTING` | fires the "being printed" email |
  | `com.prodigi.order.status.stage.changed#Complete` | `SHIPPED` | order done/dispatched |
  | `com.prodigi.order.shipments.shipment#Dispatched` | `SHIPPED` | carries `data.order.shipments[].tracking` |
  | `com.prodigi.order.shipments.shipment#Delivered` | `DELIVERED` | |
  | `com.prodigi.order.status.stage.changed#Cancelled` | `CANCELLED` | terminal |

- **Payload field paths used:** order id at `data.order.id`; tracking at
  `data.order.shipments[0].tracking.{number,carrier}`. // UNVERIFIED.
- **Mapping + transition:** the route does **no** transition logic — `mapProdigiEventToStatus`
  parses the event into the provider-agnostic `{ providerOrderId, status, tracking }` shape and
  hands it (with the token-resolved `FulfillmentOrder`) to the shared seam
  `applyFulfillmentTransition` (`src/lib/fulfillment/status.ts`), the SAME seam the Teemill polling
  path feeds (US-MFTF-14.2). The seam owns the monotonic guard, idempotency, and the buyer
  lifecycle email (US-MFTF-14.3).
- No env var needed — the per-order `webhookToken` is generated and stored on each
  `FulfillmentOrder`. `NEXT_PUBLIC_BASE_URL` must be the externally-reachable per-env host.

## Cotton-standard note

Prodigi's `paperType` attribute reveals fabric composition. The blank used to validate this integration (Bella + Canvas 1010) is **52% cotton / 48% polyester** — a blend, which violates the brand's **100%-cotton non-negotiable**. Prodigi has **no organic / 100%-cotton apparel** offering, so Prodigi apparel is a **reference integration only** (proving the `DESIGNED`/provider abstraction); production apparel is Teemill (GOTS organic cotton). If Prodigi apparel ever ships for real, add a cotton-standard guard at product-type creation that reads `paperType`.

## Key differences from Teemill

| Concern | Prodigi | Teemill |
|---|---|---|
| Sandbox | **Yes** (`api.sandbox.prodigi.com`) | None — MSW only |
| Auth | `X-API-Key` header | raw key in `Authorization` (no Bearer) + `?project=` |
| Order creation | **Single-step** `POST /orders` | Two-step `POST /orders` → `POST /orders/{id}/confirm` |
| Currency | Quote currency requestable → **USD** | **GBP only** (FX handled out-of-band) |
| Shipping cost | Real per-tier amounts (Budget/Standard/Express/Overnight) | Bundled into item cost → typically **£0** |
| Product id | SKU string (+ `attributes` for apparel) | `variantRef` URL |
| Colour data | name only, no hex | name **+ hex** |
| Shipment status | webhook path (we read the order GET) | polling `GET /orders/{ref}` (webhooks unconfirmed) |
| Fabric standard | blends incl. (no organic option) | all GOTS organic cotton |
